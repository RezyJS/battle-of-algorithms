# Алгоритмы проекта «Битва алгоритмов»

В проекте реализованы две категории алгоритмов:

1. **Системные алгоритмы** — используются платформой (генерация карт, слияние полей, исполнение скриптов)
2. **Пользовательские шаблоны** — предустановленные алгоритмы навигации операторов

---

## Часть I. Системные алгоритмы

---

### 1. Генерация лабиринта (Recursive Backtracking / DFS)

**Файл:** `src/entities/field/model/map-generator.ts`

#### Принцип работы

Алгоритм создаёт лабиринт на сетке размером W×H с помощью рекурсивного обхода в глубину (DFS):

1. **Инициализация**: вся сетка заполняется стенами (`FieldContent.WALL`)
2. **Рекурсивное прокладывание путей**:
   - Начинаем с клетки (0, 0)
   - Помечаем текущую клетку как пустую
   - Случайно перемешиваем список направлений (вверх, вниз, влево, вправо)
   - Для каждого направления: если клетка через 2 позиции не посещена — прокладываем путь (делаем пустой и промежуточную клетку, и целевую), рекурсивно продолжаем из целевой
3. **Открытие дополнительных проходов**: после генерации основного лабиринта 30% оставшихся стен (кроме граничных) удаляются, если у них 2–3 пустых соседа. Это создаёт альтернативные маршруты и делает лабиринт менее «коридорным»
4. **Размещение объектов**: на пустые клетки помещаются 2 точки спавна, 2 ключа и выход
5. **Верификация проходимости**: общая функция `validateReachability()` проверяет, что каждый спавн может достичь каждого ключа и выхода. Если нет — повторяем размещение (до 100 попыток)

#### Псевдокод

```
function generateMap(W, H):
    grid ← заполнить W×H стенами

    function carve(x, y):
        grid[y][x] ← EMPTY
        для каждого (dx, dy) в shuffle([(0,-2), (0,2), (-2,0), (2,0)]):
            nx, ny ← x+dx, y+dy
            если (nx, ny) в границах и не посещена:
                grid[y + dy/2][x + dx/2] ← EMPTY   // убрать стену между
                carve(nx, ny)

    carve(0, 0)

    // Открыть дополнительные проходы
    для каждой стены (x, y) в grid:
        если random() < 0.3 и 2 ≤ пустые_соседи ≤ 3:
            grid[y][x] ← EMPTY

    // Разместить объекты с проверкой BFS
    повторять:
        spawn1, spawn2 ← случайные далёкие пустые клетки
        key1, key2, exit ← случайные свободные клетки
    пока не (BFS(spawn1→key1) ∧ BFS(spawn1→key2) ∧ BFS(spawn2→key1) ∧ BFS(spawn2→key2) ∧ BFS(spawn1→exit) ∧ BFS(spawn2→exit))

    return grid, spawn1, spawn2
```

#### Сложность

| Этап | Временная сложность | Пространственная сложность |
|---|---|---|
| DFS-прокладывание | O(W × H) | O(W × H) — стек рекурсии + массив `visited` |
| Открытие проходов | O(W × H) | O(1) |
| BFS-верификация (одна) | O(W × H) | O(W × H) — очередь + множество `seen` |
| Полная верификация (6 BFS) | O(6 × W × H) = O(W × H) | O(W × H) |
| **Итого (1 попытка)** | **O(W × H)** | **O(W × H)** |

Worst case: до 100 попыток размещения → O(100 × W × H), но на практике корректное размещение находится за 1–3 попытки, т.к. DFS-лабиринт связный.

#### Аргументация корректности

1. **Связность**: DFS recursive backtracking гарантирует, что все клетки с чётными координатами связаны (это свойство алгоритма — он обходит все клетки ровно один раз)
2. **Проходимость**: BFS-верификация явно проверяет достижимость всех целей от обоих спавнов. Размещение повторяется, пока все 6 проверок не пройдут
3. **Разнообразие**: случайное перемешивание направлений (`shuffle`) на каждом шаге DFS + случайное удаление 30% стен даёт уникальные карты при каждой генерации

### 2. BFS-проверка проходимости

**Файл:** `src/entities/field/model/field-validation.ts` (функция `validateReachability`)

#### Принцип работы

Поиск в ширину (Breadth-First Search) от каждого спавна до набора целей:

1. Для текущего спавна инициализируем очередь с начальной клеткой и множество посещённых
2. Извлекаем клетку из головы очереди
3. Добавляем всех непосещённых проходимых соседей в очередь
4. После обхода проверяем, что все целевые клетки входят в множество посещённых
5. Повторяем это для каждого спавна; при первом недостижимом объекте возвращаем ошибку

#### Сложность

| | Сложность |
|---|---|
| Временная | O(S × (V + E)) = O(S × W×H), где S — число спавнов |
| Пространственная | O(W×H) — очередь + множество `seen` |

#### Аргументация корректности

BFS гарантированно перечисляет все клетки связной компоненты стартовой позиции в невзвешенном графе. Следовательно, любая цель достижима тогда и только тогда, когда она присутствует в множестве посещённых после завершения обхода. Эта же функция переиспользуется и генератором, и конструктором карт.

### 3. Слияние полей (mergeFields)

**Файл:** `src/entities/field/model/field-utils.ts`

#### Принцип работы

Функция создаёт визуальное представление поля для конкретного шага:

1. Копирует статическую карту (базовый слой)
2. Для каждого оператора берёт его позицию из истории на данном шаге
3. Если оператор не вышел — помещает его UID на карту
4. Если клетка содержит ключ/выход — конкатенирует (например, `key1_🔴`)

#### Сложность

| | Сложность |
|---|---|
| Временная | O(W×H + P), где P — количество операторов (обычно 2) |
| Пространственная | O(W×H) — копия карты |

### 4. Исполнение пользовательского кода (executeScript)

**Файл:** `src/features/battle-runner/model/execute-script.ts`

#### Принцип работы

1. Создаёт обёртку над методами оператора, которая на каждый вызов движения инкрементирует счётчик шагов
2. При превышении `maxSteps` (1000) бросает исключение `'Max steps reached'`
3. Создаёт JavaScript-функцию через `new Function('operator', scriptString)` и вызывает её с обёрнутым оператором
4. Перехватывает все ошибки и возвращает текст ошибки или `null` при успехе

#### Сложность

Зависит от пользовательского скрипта. Верхняя граница: O(maxSteps × C), где C — стоимость одной операции оператора (O(1) для движения, O(r²) для `scan(r)`, O(W×H) для `getDistanceTo`).

---

### 5. Манхэттенское расстояние (getDistanceTo)

**Файл:** `src/entities/operator/model/operator.ts`

#### Принцип работы

Перебирает все клетки статической карты, находит первую клетку указанного типа (`key*` или `exit`) и возвращает манхэттенское расстояние: `|x₁ - x₂| + |y₁ - y₂|`.

#### Сложность

| | Сложность |
|---|---|
| Временная | O(W × H) — полный перебор карты |
| Пространственная | O(1) |

**Примечание**: это эвристическое расстояние, а не длина реального пути. Манхэттенское расстояние является нижней оценкой реального пути в лабиринте (admissible heuristic), что делает его пригодным для A*-подобных стратегий.

---

### 6. Сканирование области (scan)

**Файл:** `src/entities/operator/model/operator.ts`

#### Принцип работы

Возвращает 2D-массив размером `(2r+1) × (2r+1)` с содержимым клеток в радиусе `r` от оператора. Клетки за пределами карты помечаются как `'out_of_bounds'`.

#### Сложность

| | Сложность |
|---|---|
| Временная | O((2r+1)²) = O(r²) |
| Пространственная | O(r²) — возвращаемый массив |

## Часть II. Пользовательские алгоритмы навигации (шаблоны)

Все шаблоны расположены в `src/features/script-editor/model/templates.ts`.

### 1. Простой обход (Simple Explorer)

#### Принцип работы

Детерминированный обход с фиксированным приоритетом направлений: вправо → вниз → влево → вверх. На каждом шаге выбирает первое доступное направление.

```
на каждом шаге:
    если можно вправо → идти вправо
    иначе если можно вниз → идти вниз
    иначе если можно влево → идти влево
    иначе если можно вверх → идти вверх
```

#### Сложность

- **Временная**: O(N), где N — количество итераций (до 800). Каждая итерация — O(1) (4 проверки + 1 ход).
- **Пространственная**: O(1) — никаких дополнительных структур данных.

#### Анализ корректности

**Не гарантирует нахождение выхода.** Алгоритм может зациклиться в тупиках — например, в U-образном коридоре оператор будет бесконечно перемещаться между двумя клетками. Подходит только для простых открытых карт.

**Класс**: жадный алгоритм без памяти.

---

### 2. Правило правой руки (Wall Follower)

#### Принцип работы

Классический алгоритм выхода из лабиринта. Оператор всегда «прижимается» к правой стене:

- Хранит текущее направление `dir` (0=вправо, 1=вниз, 2=влево, 3=вверх)
- На каждом шаге пытается повернуть вправо (относительно текущего направления), если невозможно — идёт прямо, если и прямо нельзя — поворачивает влево, и наконец разворачивается

```
на каждом шаге:
    если можно повернуть вправо → повернуть и пройти
    иначе если можно прямо → пройти
    иначе если можно повернуть влево → повернуть и пройти
    иначе → развернуться
```

#### Сложность

- **Временная**: O(N), N ≤ 800. Каждая итерация — O(1).
- **Пространственная**: O(1) — одна переменная `dir`.

#### Анализ корректности

**Гарантирует выход из односвязного лабиринта** (без островов). Если лабиринт имеет изолированные стены (острова), оператор может зациклиться, обходя остров бесконечно. Это следует из теоремы о том, что правило руки работает только для simply-connected mazes.

Для наших сгенерированных карт (DFS + дополнительные проходы) лабиринт может быть не односвязным, поэтому корректность **не гарантирована** на 100% карт.

**Класс**: детерминированный алгоритм с конечной памятью (O(1)).

---

### 3. Зигзаг (Zigzag Pattern)

#### Принцип работы

Движение змейкой по горизонтали:

1. Двигается вправо, пока может
2. Когда упирается — сдвигается на одну клетку вниз (или вверх) и меняет направление на влево
3. Двигается влево, пока может
4. Снова сдвигается вниз и меняет на вправо
5. Повторяет

```
goingRight ← true
на каждом шаге:
    если goingRight:
        если можно вправо → идти вправо
        иначе → сдвиг вниз/вверх, goingRight ← false
    иначе:
        если можно влево → идти влево
        иначе → сдвиг вниз/вверх, goingRight ← true
```

#### Сложность

- **Временная**: O(N), N ≤ 800. Каждая итерация — O(1).
- **Пространственная**: O(1) — один булев флаг.

#### Анализ корректности

**Не гарантирует нахождение выхода.** Эффективен на открытых картах, но в лабиринтах с узкими коридорами может пропускать целые области. Зигзаг обеспечивает систематическое покрытие прямоугольных областей, но не адаптируется к топологии лабиринта.

**Класс**: систематический обход без памяти.

---

### 4. Умный разведчик (Smart Explorer)

#### Принцип работы

Использует **систему меток** (breadcrumbs) для отслеживания посещённых клеток:

1. Помечает текущую клетку при каждом шаге
2. При выборе направления приоритизирует **непосещённые** клетки
3. Если все соседние клетки посещены — выбирает любую проходимую (с возможным возвратом)

```
на каждом шаге:
    пометить текущую клетку

    для каждого направления (право, низ, лево, верх):
        если проходимо И не помечено → идти туда, стоп

    если не удалось:
        для каждого направления:
            если проходимо → идти туда, стоп
```

#### Сложность

- **Временная**: O(N), N ≤ 800. Каждая итерация — O(4) проверок `isMarked` + O(1) `markCell`.
- **Пространственная**: O(V), где V — количество посещённых клеток (до W×H). Хранит Set строк `"x,y"`.

#### Анализ корректности

**Высокая вероятность нахождения выхода**, но **не гарантирована**. Алгоритм реализует упрощённый DFS без явного стека возврата. В ситуации, когда все соседи помечены, он выбирает произвольного соседа (не обязательно ведущего к новым областям). Полноценный DFS с backtracking-стеком гарантировал бы полный обход.

Тем не менее, на практике этот алгоритм значительно эффективнее простого обхода, т.к. избегает повторного посещения клеток в большинстве случаев.

**Класс**: жадный алгоритм с памятью O(V).

---

### 5. Искатель цели (Target Seeker)

#### Принцип работы

Использует **манхэттенское расстояние** для выбора направления:

1. Определяет текущую цель: ключ (если нет ключа) или выход (если есть)
2. На каждом шаге пробует все направления
3. Выбирает первое доступное (в данной реализации — без оптимизации по расстоянию)

```
на каждом шаге:
    цель ← "exit" если есть ключ, иначе "key"
    расстояние ← getDistanceTo(цель)

    попробовать все направления, идти в первое доступное
```

#### Сложность

- **Временная**: O(N × W × H), т.к. `getDistanceTo` на каждом шаге перебирает всю карту.
- **Пространственная**: O(1).

#### Анализ корректности

**Не гарантирует оптимальный путь.** Текущая реализация использует `getDistanceTo` для информации, но выбор направления — просто первое доступное. Для полноценного «движения к цели» нужно сравнивать расстояния после каждого потенциального хода.

Манхэттенское расстояние — допустимая эвристика (admissible), т.е. оно никогда не переоценивает реальное расстояние в лабиринте. Это свойство делает его корректным для A*-подобных алгоритмов.

**Класс**: эвристический алгоритм.

### 6. Сканер (Scanner)

#### Принцип работы

Самый продвинутый шаблон. Использует **сканирование области** для принятия решений:

1. Сканирует окрестность радиуса 2 (матрица 5×5)
2. Ищет в области сканирования цель (ключ или выход)
3. Если цель найдена — движется в её сторону
4. Если не найдена — движется случайно

```
на каждом шаге:
    area ← scan(2)     // матрица 5×5
    цель ← "exit" если есть ключ, иначе "key"

    для каждой клетки в area:
        если клетка = цель:
            вычислить смещение (dx, dy)
            идти в сторону цели
            стоп

    если цель не найдена:
        случайное движение
```

#### Сложность

- **Временная**: O(N × r²), где r = 2, итого O(N × 25) = O(N). Каждая итерация: O(r²) на сканирование + O(1) на движение.
- **Пространственная**: O(r²) — массив сканирования.

#### Анализ корректности

**Не гарантирует нахождение выхода.** При случайном блуждании вне зоны видимости оператор может долго не найти цель. Однако комбинация направленного движения (когда цель видна) и случайного блуждания (когда не видна) на практике даёт хорошие результаты — это фактически **случайное блуждание с локальной оптимизацией**.

Теоретически, случайное блуждание на конечном связном графе посетит каждую вершину с вероятностью 1 (за ожидаемое время O(V²)), но на практике 800 шагов может не хватить для больших карт.

**Класс**: стохастический алгоритм с локальной эвристикой.

## Часть IV. Рекомендации по улучшению

Для пользователей, желающих написать собственный алгоритм:

1. **DFS с backtracking**: хранить стек посещённых клеток и возвращаться по нему при тупике → гарантирует полный обход
2. **BFS от текущей позиции**: после получения ключа запустить BFS к выходу (если API позволяет предрассчёт)
3. **A\*-эвристика**: использовать `getDistanceTo` как h(n) и минимизировать f(n) = g(n) + h(n) на каждом шаге
4. **Комбинация**: Умный разведчик + Сканер — метки для глобального обхода, scan для локальной навигации к видимым целям
