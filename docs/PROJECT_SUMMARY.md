# Структурированная информация по проекту

## 1. Описание проекта

- Веб-платформа `Battle of Algorithms` для соревнования пользовательских алгоритмов навигации в лабиринте.
- По коду и документации продукт совмещает 2 части:
  - клиентскую симуляцию боя на фронтенде;
  - серверный контур для авторизации, отправки кода, модерации и выбора активного боя.
- Основная задача:
  - дать пользователю написать алгоритм на JavaScript;
  - отправить код на модерацию;
  - запускать бой между двумя одобренными решениями на арене.
- Целевая аудитория:
  - участники, которые пишут алгоритмы;
  - модераторы и администраторы, которые проверяют код и управляют ареной;
  - по `docs/PROJECT.md` проект позиционируется как образовательная платформа для изучения алгоритмов навигации.

## 2. Функциональность

- **Пользователь**
  - Просмотр текущего активного боя на главной странице `/`.
  - Просмотр пошаговой симуляции: старт, пауза, сброс, скорость, лог событий, результат.
  - Редактирование своего алгоритма на странице `/editor`.
  - Сохранение черновика отправки.
  - Отправка кода на модерацию.
  - Просмотр статуса своей последней отправки, версии и комментария модератора.
  - Просмотр профиля `/profile`.
  - Вход и выход через Keycloak.
- **Модератор / админ**
  - Всё, что доступно пользователю.
  - Доступ к панели модерации `/moderation`.
  - Просмотр списка всех отправок.
  - Смена статуса отправки: `under_review`, `approved`, `rejected`, `returned`.
  - Доступ к управлению составом арены `/moderation/arena`.
  - Выбор левого и правого игрока из пользователей с `approved` отправками.
  - Снятие текущего активного боя.
  - Доступ к конструктору карт `/map-editor`.
- **Игровая логика фронтенда**
  - Режимы `race` и `duel`.
  - Генерация случайной карты.
  - Ручное создание карты в конструкторе.
  - Валидация карты по достижимости BFS.
  - Выполнение пользовательского JS-скрипта с ограничением по шагам.

## 3. Архитектура

- Общая структура:
  - `Next.js 16` приложение в корне репозитория.
  - `FastAPI` backend в директории `backend/`.
  - `PostgreSQL` и `Keycloak` в `infra/docker-compose.yml`.
- Взаимодействие частей:
  - Пользователь аутентифицируется через Keycloak по OIDC.
  - Frontend получает code callback, синхронизирует профиль в backend и создаёт серверную cookie-сессию `boa_session`.
  - Серверные страницы Next.js и server actions ходят в backend по внутренним endpoint’ам.
  - Backend не обслуживает браузер напрямую для пользовательских действий; он используется как внутренний API для Next.js.
  - Симуляция боя выполняется на клиенте через Zustand и игровые сущности, а не на backend.
- Основные модули frontend:
  - `app/` — маршруты и server actions.
  - `src/app/model/` — Zustand stores.
  - `src/features/` — редактор, исполнение алгоритмов, auth UI.
  - `src/entities/` — `Operator`, карта, генерация, валидация.
  - `src/widgets/` — игровое поле, control panel, event log, map editor.
- Основные модули backend:
  - `api/routes/` — внутренние маршруты auth, submissions, moderation, arena, health.
  - `services/` — бизнес-логика пользователей, отправок, боёв.
  - `models/` — SQLAlchemy-модели.
  - `schemas/` — Pydantic-схемы.
  - `alembic/` — миграции схемы.

## 4. Технологии

- **Frontend**
  - `Next.js 16` + App Router.
  - `React 19`.
  - `TypeScript 5`.
  - `Zustand`.
  - `Tailwind CSS 4`.
  - `CodeMirror` через `@uiw/react-codemirror`.
  - `lucide-react`.
  - `arctic` для OIDC/Keycloak.
- **Backend**
  - `FastAPI`.
  - `SQLAlchemy 2`.
  - `Pydantic Settings`.
  - `Alembic`.
  - `psycopg`.
  - `uvicorn`.
- **База данных**
  - `PostgreSQL 16`.
- **Инфраструктура**
  - `docker-compose` для `postgres` и `keycloak`.
  - Инициализация отдельной БД Keycloak через `infra/postgres/init/01-create-keycloak-db.sql`.
  - Frontend и backend запускаются локально отдельно.
  - CORS настроен на `http://localhost:3000`.

## 5. Реализация (технические детали)

- **Фронтенд**
  - Архитектура организована в стиле FSD: `app -> widgets -> features -> entities -> shared`.
  - Глобальное состояние игры и редактора карты хранится в двух Zustand store:
    - `game-store.ts`
    - `map-editor-store.ts`
  - Основной UI строится на server pages + client components.
- **Работа с API**
  - Вызовы backend инкапсулированы в `src/shared/lib/api/internal.ts`.
  - Все запросы идут с заголовком `x-internal-api-secret`.
  - Используются server-side fetch и server actions, а не клиентские запросы из браузера.
- **Авторизация / аутентификация**
  - OIDC через Keycloak.
  - `app/api/auth/login` и `register` создают auth request.
  - `app/api/auth/callback` валидирует `state`, меняет code на токены, синхронизирует пользователя в backend.
  - Сессия приложения хранится в подписанной `httpOnly` cookie.
  - Роли берутся из `realm_access.roles`, допустимые роли: `user`, `moderator`, `admin`.
- **Управление состоянием**
  - `game-store` хранит:
    - скрипты игроков;
    - историю состояний;
    - текущий шаг;
    - режим боя;
    - карту;
    - результат;
    - скорость проигрывания;
    - ошибку скрипта.
  - `map-editor-store` хранит:
    - сетку;
    - размеры;
    - активную кисть;
    - ошибку валидации.
- **Формы / валидация**
  - Редактор:
    - отправка доступна только если код не пустой и симуляция не запущена;
    - отдельной схемной валидации формы нет.
  - Модерация:
    - server action проверяет роль и обязательные поля формы.
  - Состав арены:
    - server action требует 2 player id;
    - backend дополнительно проверяет, что игроки разные, существуют и у обоих есть `approved` отправки.
  - Конструктор карт:
    - проверяет наличие 2 spawn, минимум 1 key, 1 exit;
    - периметр обязан быть стеной;
    - BFS проверяет достижимость целей от обоих спавнов.
- **Исполнение пользовательского кода**
  - Скрипт выполняется через `new Function('operator', scriptString)`.
  - Пользователю передаётся ограниченный объект `operator`.
  - На шаги наложен лимит `1000`.
  - Ошибки выполнения возвращаются в UI как `scriptError`.

## 6. Основные пользовательские сценарии

- **1. Вход в систему**
  - Пользователь нажимает вход.
  - Frontend редиректит в Keycloak.
  - После callback создаётся локальная cookie-сессия.
  - Профиль синхронизируется в backend таблицу `users`.
- **2. Работа с кодом**
  - Авторизованный пользователь открывает `/editor`.
  - Видит свою последнюю отправку.
  - Редактирует код в CodeMirror.
  - Сохраняет черновик или отправляет код на модерацию.
- **3. Модерация отправки**
  - Модератор открывает `/moderation`.
  - Просматривает список отправок и код.
  - Пишет комментарий.
  - Меняет статус отправки.
  - Backend создаёт запись в `submission_reviews` и `audit_log`.
- **4. Назначение активного боя**
  - Модератор открывает `/moderation/arena`.
  - Выбирает двух игроков с одобренными решениями.
  - Сохраняет пару.
  - Backend закрывает предыдущий `active` бой и создаёт новый.
  - Главная страница начинает показывать выбранные скрипты и участников.
- **5. Просмотр боя**
  - Пользователь открывает `/`.
  - Frontend получает активный бой из backend.
  - Коды двух игроков записываются в Zustand.
  - Локально запускается симуляция и пошаговое воспроизведение.

## 7. Интерфейс

- `/`
  - Арена.
  - Показывает текущий бой или сообщение, что бой не выбран.
  - Отображает игровое поле, панель управления, лог событий, легенду.
- `/editor`
  - Страница “Мой код”.
  - Показывает статус, версию, кнопки сохранить/отправить, CodeMirror, справку по API оператора, советы.
- `/profile`
  - Профиль пользователя.
  - Показывает имя, username, email, роли и ссылки на доступные разделы.
- `/rules`
  - Правила игры.
  - Описывает цель, режимы, условия победы, очки, ограничения.
- `/map-editor`
  - Конструктор карт.
  - Доступен только `moderator/admin`.
  - Позволяет рисовать карту, менять размеры, валидировать и применять её в игру.
- `/moderation`
  - Панель модерации.
  - Список отправок, код, статус, даты, комментарий, кнопки смены статуса.
- `/moderation/arena`
  - Управление составом арены.
  - Выбор двух игроков, просмотр наличия `approved`, текущая активная пара.
- `/login`
  - Служебный редирект в auth flow.
- `/register`
  - Сейчас редиректит в `/api/auth/login`, а не в `/api/auth/register`.
- `/auth-error`
  - Экран ошибок авторизации.
- `/forbidden`
  - Экран отсутствия прав.

## 8. Особенности и решения

- Frontend и backend разделены, но пользовательский бой считается локально на фронтенде.
- Для backend используется только внутренний API с `x-internal-api-secret`; прямой публичный пользовательский REST-контур не реализован.
- Локальная сессия приложения подписывается HMAC, а не хранится через внешнюю session library.
- При выборе активного боя backend автоматически завершает все предыдущие `ACTIVE` бои.
- Для арены выбираются именно последние `approved` отправки каждого пользователя.
- Конструктор карт не просто проверяет форму карты, а валидирует достижимость через BFS.
- Генератор карты делает лабиринт DFS + дополнительные проходы и затем отдельно проверяет достижимость ключей и выхода.
- В `duel` у оператора появляется доступ к `getOpponentPosition()`.
- Есть аудит действий:
  - смена статуса отправки;
  - назначение активного боя;
  - снятие активного боя.

## 9. Текущий статус

- **Полностью готово**
  - Frontend-маршруты: арена, редактор, профиль, правила, модерация, состав арены, конструктор, auth-error, forbidden.
  - Локальная игровая симуляция и визуализация боя.
  - Генерация карты и ручной конструктор с валидацией.
  - OIDC-вход через Keycloak callback flow.
  - Локальная cookie-сессия и роль-ориентированный доступ на страницах.
  - Backend-модели, миграции и внутренние API для:
    - sync user;
    - own submission;
    - moderation submissions;
    - active battle;
    - arena users.
  - Базовый seed для пользователей, отправок и активного боя.
- **Частично реализовано**
  - Регистрация:
    - маршрут `app/api/auth/register` есть;
    - страница `/register` сейчас редиректит на логин, а не на регистрацию.
  - Авторизация на backend:
    - есть placeholder `/api/protected`;
    - проверки JWT/ролей на backend нет, вместо этого используется внутренний секрет.
  - Боевая серверная логика:
    - backend хранит активный бой и коды;
    - сам расчёт боя остаётся на фронтенде.
  - Работа с отправками:
    - пользователь видит только последнюю отправку;
    - отдельной страницы истории отправок нет.
- **Отсутствует**
  - Публичный backend API для клиента с пользовательской авторизацией по токенам.
  - Реальная серверная валидация/исполнение пользовательского кода.
  - История боёв, результаты боёв и серверное сохранение исходов.
  - Отдельная админ-функциональность сверх прав модератора.
  - Прокси, production-deploy конфиги, CI/CD конфиги в репозитории не найдены.
  - Формальная валидация форм через schema library на фронтенде не используется.
