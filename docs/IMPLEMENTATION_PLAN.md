# План внедрения auth, moderation, backend и БД

## Цель

Перевести текущий фронтовый прототип в полноценное приложение с:

- авторизацией через Keycloak;
- серверными сессиями;
- ролями `user` / `moderator` / `admin`;
- backend на Python;
- PostgreSQL;
- модерацией отправленного кода;
- отображением текущего боя;
- персональной страницей отправки кода и статусами модерации.

## Текущее состояние проекта

Сейчас в репозитории есть только Next.js-приложение с клиентским прототипом:

- арена `/`;
- редактор `/editor`;
- конструктор карты `/map-editor`;
- правила `/rules`;
- локальное состояние на Zustand;
- нет backend API;
- нет БД;
- нет auth и сессий;
- нет серверной модели пользователей, боёв и отправок кода.

## Выбранная целевая архитектура

### Frontend

- `Next.js 16` остаётся основным веб-приложением.
- Добавляем защищённые маршруты и серверные проверки сессии.
- Логин и регистрация доступны на сайте через наши страницы `/login` и `/register`.
- Источник истины по идентификации пользователей: `Keycloak`.

### Auth и сессии

- Используем `Keycloak` как Identity Provider по `OIDC`.
- Храним сессию приложения в `httpOnly secure cookies`.
- Роли и права берём из Keycloak claims / realm roles.
- Базовые роли:
  - `user`
  - `moderator`
  - `admin`

### Backend

- Выносим backend в отдельную директорию `backend/`.
- Стек:
  - `FastAPI`
  - `SQLAlchemy`
  - `Alembic`
  - `psycopg`
  - `Pydantic`
- Backend отвечает за:
  - API для текущего боя;
  - API отправки кода;
  - API модерации;
  - хранение и выдачу статусов;
  - работу с PostgreSQL;
  - серверные проверки ролей.

### База данных

- `PostgreSQL` как основная БД.
- Все боевые сущности, отправки кода, статусы, выбранные игроки и история модерации храним в Postgres.

### Dev-инфраструктура

- Поднимаем локально через `docker compose`:
  - `postgres`
  - `keycloak`
- Сам frontend и backend можно запускать локально без контейнеризации на этапе разработки.

## Основные сущности БД

### Пользователи

- `users`
- поля:
  - `id`
  - `keycloak_user_id`
  - `username`
  - `email`
  - `display_name`
  - `is_active`
  - `created_at`
  - `updated_at`

Примечание: основная identity-модель живёт в Keycloak, но в Postgres нам нужен локальный профиль и связка с боевыми сущностями.

### Бои

- `battles`
- поля:
  - `id`
  - `title`
  - `status` (`draft`, `scheduled`, `active`, `finished`)
  - `left_player_id`
  - `right_player_id`
  - `map_config`
  - `started_at`
  - `finished_at`
  - `created_by`
  - `updated_at`

### Отправки кода

- `code_submissions`
- поля:
  - `id`
  - `user_id`
  - `battle_id` nullable
  - `code`
  - `language`
  - `status` (`draft`, `submitted`, `under_review`, `approved`, `rejected`, `returned`)
  - `submitted_at`
  - `moderated_at`
  - `moderated_by`
  - `moderation_comment`
  - `version`

### История модерации

- `submission_reviews`
- поля:
  - `id`
  - `submission_id`
  - `action`
  - `comment`
  - `created_by`
  - `created_at`

### Аудит действий

- `audit_log`
- для критичных действий:
  - назначение текущего боя;
  - смена игроков;
  - модерация кода;
  - изменение ролей на стороне приложения, если понадобится.

## Статусы отправки кода

Минимальный набор статусов, который нужно реализовать:

- `draft` — черновик пользователя
- `submitted` — отправлено на модерацию
- `under_review` — модератор взял в работу
- `approved` — одобрено
- `rejected` — отклонено
- `returned` — возвращено на доработку

## Матрица доступа

### User

- логин / логаут / регистрация;
- просмотр текущего боя;
- просмотр заглушки, если бой не выбран;
- доступ к своей странице отправки кода;
- создание и повторная отправка своего кода;
- просмотр только своих статусов и истории отправок.

### Moderator

- всё, что может `user`;
- доступ к панели модерации;
- просмотр списка отправок;
- изменение статусов отправок;
- доступ к конструктору карт;
- выбор активного боя;
- выбор игроков для текущего боя.

### Admin

- всё, что может `moderator`;
- управление правами и административными настройками, если понадобится;
- доступ ко всем модерационным операциям.
